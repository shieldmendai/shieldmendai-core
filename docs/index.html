The product

Recovery infrastructure built around control.

ShieldMendAi separates observation, authorization, action planning, verification, rollback, and incident reporting. The architecture is designed to resist broad permissions, repeated repair loops, and unverified “fixes.”

01 / OBSERVE

Normalize failure signals

Typed findings for services, timers, processes, APIs, ports, executables, and files—currently supplied through safe scenarios and fixture-confined checks.

02 / AUTHORIZE

Deny by default

Exact target and action allowlists, approval policies, risk classes, preconditions, verification plans, and rollback requirements gate every simulated repair.

03 / RECOVER

Bound every attempt

Finite retries, cooldowns, backoff, circuit breakers, duplicate suppression, and manual escalation are modeled before any production execution exists.

How it works

One explicit lifecycle. No hidden leap to automation.

Each boundary is independently typed and auditable. Today, the entire lifecycle consumes controlled inputs and returns deterministic simulated outcomes.

What exists—and what does not.

Clear capability labeling is part of the product. Simulation is not described as deployment, and architecture is not described as live operation.

Implemented and tested

  • Standalone Python package and command-line interface
  • Typed configuration, finding, repair, recovery, incident, and notification models
  • Deny-by-default authorization and exact allowlists
  • Bounded recovery, circuit breakers, and loop protection
  • Versioned incident records and integrity validation
  • 141 verified passing automated tests

Production unavailable

  • Live system or process observation
  • Service restarts or real repair execution
  • Production incident persistence or deletion
  • Real Telegram, email, SMS, webhook, or local delivery
  • Customer deployment or autonomous production repair
  • Vulnerability scanning or universal compatibility
